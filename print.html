Third-party native code in crates

Introduction

This article investigates a specific type of crates, those including third-party code (often C/C++ libraries) statically linked into Rust binaries.

The goal is to explore the current situation and the challenges, and then to propose some improvements and proof-of-concepts for dedicated tooling.

Source crates

A "source crate" is a crate that includes third party sources that get compiled into the resulting library (or binary).

There are several types of them. Some are dedicated crates (usually containing -src in the name), some are -sys crates which are able to compile the library they are providing an interface for, either by default or when enabled by a feature flag.

A good explanation of how to include a static dependency build in a -sys crate is available in this article.

Widely used examples are openssl-src or curl-sys.

Sometimes the code is copied into the repository, sometimes it is configured as a git submodule. In any case the source becomes part of the the crate uploaded to the registry.

Goals

Several goals:

  • Explore third-party-code in crates (through the lens of submodules in most popular crates)
  • Give leads to improve management of -src crates in the ecosystem
  • Proof-of-concept of source crates database and integration to the ecosystem

Exploration

Goals

In order to understand the current situation, besides ... -src crates, a good option it to analyze the most downloaded crates for third-party code.

  • Get a list of the 10% most downloaded crates
  • Try to clone their documented repository
  • Look for submodules

This misses:

  • 90% of the crates
  • All code that is simply copied and not configured as submodule

But it should give a representative enough overview.

Methodology

  • Selected all crates with more than 10k downloads on the 2021-05-05 (6984 crates)
  • Tried to clone all repositories associated (6322 crates successfully clone)
  • Look for submodules in the cloned crates, 592 crates contained submodules

Among those 595 crates containing submodules, there were 251 different submodule repositories (after excluding submodules in the same github organization as the source repository).

NOTE : when several crates are hosted in the same directory, the submodule appears several time, which explain the high number of duplicates.

These 251 repositories were:

  • 89 contained data (unicode data, time-zones, syntax highlighting definitions, etc.)
  • 32 contained tests suites or test data set
  • 119 libraries, including 13 Rust libraries, 24 C++ libraries and 71 C libraries

(these numbers are based on approximate tagging in repositories.toml but give an estimate).

In short, there are 95 C/C++ native libraries included in the top 7k crates using submodules, not counting those directly copied into the crate source.

Found issues

  • Three crates use code from the https://github.com/copies organization which provides mirrors for projects hosted outside of Github. It gives no indication about who it's run by, and seem to be updated regularly, nor following upstream versions.
  • Some crates use http:// URLs in submodules.

General problems

These third-party dependencies are often invisible, sometimes embedded by default. They often don't have proper security advisories, and random upstream update.

Perspectives

  • trust (a la cargo-supply-chain) (not the goal here)

  • security (will I be aware of vulnerabilities?)

  • upgrades with static libraries

Several options:

  • Try to encourage adoption of -src crates, with a proper versioning convention. This could start in cargo documentation.
  • Try to crate a database of crates including source code with the features controlling its inclusion to be able to use the information, to be able to manage non dedicated source crates.

This would allow:

  • (cargo-audit)[https://github.com/Shnatsel/rust-audit#what-about-keeping-track-of-versions-of-statically-linked-c-libraries]

Current issues

Source code embedded is invisible to most tooling:

  • Licenses are not visible, and for example cargo deny check licenses cannot check them. This may put users at legal risk.

  • https://github.com/alexcrichton/curl-rust/issues/321

  • Upstream versions are generally not easily understandable

  • When part of a -sys crate it is not practical to file security advisories for the whole crate for a vulnerability in the third-party code.

  • It is hard to list such crates, even from a specific Cargo.lock.

  • It is hard to keep track of the code sources (for example in a cargo supply-chain perspective)

  • It is hard to keep track of vulnerabilities that may be built into the binary

Propositions

  • Document the -src crates as best practice in cargo documentation
  • Move most used included code from -sys crates to dedicated -src crates
  • File advisories for vulnerabilities in sources (already done for openssl-src). We could automate detection based on CVEs for common libraries.
  • Document licenses and upstream repositories/provider somehow

Conclusion

We now have tooling to manage our rust dependency tree (cargo-supply-chain, cargo-deny, cargo-tree, etc.) but they are generally blind to bundled third party native code.

It is quite used though, and some -sys even use statically linked dependencies by default.

Open questions:

  • How to integrate these information into Cargo.toml and the crate itself? Especially to standardize versioning and document licenses.
  • Should -sys crates allowing building dependencies should split it to an src crate?
