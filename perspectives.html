Third-party native code in crates

Perspectives

  • trust (a la cargo-supply-chain) (not the goal here)

  • security (will I be aware of vulnerabilities?)

  • upgrades with static libraries

Several options:

  • Try to encourage adoption of -src crates, with a proper versioning convention. This could start in cargo documentation. This allows treating the source crate as if it was the upstream package for security advisories, and other tooling.
  • Try to crate a database of crates including source code with the features controlling its inclusion to be able to use the information, to be able to manage non dedicated source crates.

This would allow:

  • To track statically built dependencies in (rust-audit)[https://github.com/Shnatsel/rust-audit#what-about-keeping-track-of-versions-of-statically-linked-c-libraries]
  • To track vulnerabilities in (cargo-audit)[https://github.com/RustSec/rustsec]/(cargo-deny)[https://github.com/EmbarkStudios/cargo-deny]

Current issues

Source code embedded is invisible to most tooling:

  • Licenses are not visible, and for example cargo deny check licenses cannot check them. This may put users at legal risk.

  • https://github.com/alexcrichton/curl-rust/issues/321

  • Upstream versions are generally not easily understandable

  • When part of a -sys crate it is not practical to file security advisories for the whole crate for a vulnerability in the third-party code.

  • It is hard to list such crates, even from a specific Cargo.lock.

  • It is hard to keep track of the code sources (for example in a cargo supply-chain perspective)

  • It is hard to keep track of vulnerabilities that may be built into the binary

Propositions

  • Document the -src crates as best practice in cargo documentation
  • Move most used included code from -sys crates to dedicated -src crates
  • File advisories for vulnerabilities in sources (already done for openssl-src). We could automate detection based on CVEs for common libraries.
  • Document licenses and upstream repositories/provider somehow

Conclusion

We now have tooling to manage our rust dependency tree (cargo-supply-chain, cargo-deny, cargo-tree, etc.) but they are generally blind to bundled third party native code.

It is quite used though, and some -sys even use statically linked dependencies by default.

Open questions:

  • How to integrate these information into Cargo.toml and the crate itself? Especially to standardize versioning and document licenses.
  • Should -sys crates allowing building dependencies should split it to an src crate?
