Third-party native code in crates

Exploration

Goals

In order to understand the current situation, besides recognizable -src crates, a first step it to analyze the most downloaded crates for third-party code:

  • Get a list of the 10% most downloaded crates
  • Try to clone their documented repository
  • Look for submodules. This misses all the code that is simply copied and not configured as submodule but it should give a representative enough overview.

Methodology

  • Selected all crates with more than 10k downloads on the 2021-05-05 (6984 crates)
  • Tried to clone all repositories associated (6322 crates successfully cloned with scripts/clone.sh)
  • Look for submodules in the cloned crates, 595 crates contained submodules. The results computed with scripts/modules.py is on data/crates.json.

Among those 595 crates containing submodules, there were 251 different submodule repositories (after excluding submodules in the same github organization as the source repository).

The submodules with the repositories they are part of, computed with filter.py, is in data/submodules.json.

NOTE: when several crates are hosted in the same directory, the submodule appears several time, which explain the high number of duplicates.

Among these 251 repositories:

  • 89 contained data sets (unicode data, time zones, syntax highlighting definitions, etc.)
  • 32 contained tests suites or test data set
  • 119 libraries, including 13 Rust libraries, 24 C++ libraries and 71 C libraries

(these numbers are based on approximate tagging in data/repositories.toml but give an estimate).

In short, there are 95 C/C++ native libraries included in the top 7k crates using submodules, not counting those directly copied into the crate source.

A total of 7 crates including a C/C++ library among these have the -src in their name:

  • boringssl-src
  • penblas-src
  • sqlite3-src
  • openssl-src
  • netlib-src
  • zeromq-src
  • luajit-src

Notes

  • Three crates use code from the https://github.com/copies organization which provides mirrors for projects hosted outside of Github. It gives no indication about who it's run by, and does not seem to be updated regularly, nor following upstream versions.

General problems

These third-party dependencies are often invisible, sometimes embedded by default. They often don't have proper security advisories, and random upstream update. Sometimes upstream is not trustworthy either.
